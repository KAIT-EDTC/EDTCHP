## 環境構築
### Gitの環境構築
まずGitを[ダウンロード](https://git-scm.com/downloads/win)する  
インストール手順は全部nextを押せばOK  
Gitをインストールしたら*Git Bash*を起動して、下記のコマンドを入力  
```bash
git config --global user.name "任意の名前を入力"  
git config --global user.email "自分のメアド入力"  
```  
**これでGitコマンドを使えるようになる。(以降はコマンドプロンプトでもGit使用可)**
### ローカルリポジトリを作成(リモートからクローンする)
リポジトリを作成したいディレクトリに移動して、下記のコマンドを入力  
```bash
git clone https://github.com/fami-gb/EDTCHP.git  
```  
これでローカルのリポジトリ作成が完了したので後はvscodeで煮るなり焼くなりしよう。

## Gitのあれこれ
### ブランチ作成～リモートにプッシュするまで
- ローカルでブランチ作成して移動 `git checkout -b ブランチ名`
- 変更をステージング`git add .`or`git add ディレクトリ名`or`git add ファイル名`  
  `git add .`は**余分な差分を生む可能性があるので注意。**
- ステージングされたものをコミット`git commit -m "メッセージ"`
- コミットをリモートにプッシュ`git push origin リモートに上げたいブランチ名`

### issueを活用したブランチ作成
実はissueの右サイドの**Development**からリモートブランチを作成することが出来る。  
その際に、下記のコマンドを実行するようにダイアログが表示される。  
```bash
git fetch origin  
git checkout ブランチ名  
```  
一個目のコマンドはリモートブランチを取り込むコマンドだ。  
こうすることでわざわざローカルでブランチを切らなくても先ほど作ったブランチをそのまま使用できる。  
だから二個目のコマンドで`-b`オプションなしでブランチ移動できている。

### プッシュしてしまったコミットをなかったことにしたい
1. コミットの変更を残しつつ元に戻す  
 ```bash
git revert 戻したいコミットのハッシュ値  
```  
 このコマンドはコミットの変更をなかったことにするというコミットを作成します。
 コミットのハッシュ値取得方法は調べるか勘でいくか、聞いてください  
3. コミットの変更すら元に戻す  
   ```bash
   git reset --hard HEAD~戻したい個数  
```  
   戻したい個数はHEADからいくつ遡るかです。  
   `HEAD~3`であるならば新しいものから三つ削除します。

### コンフリクト(修正場所被り)が起きた場合
例えば、Aがbranch_Aでbase.htmlを修正し、Bもまたbranch_Bでbase.htmlを修正した場合、  
mainにマージした際にGitくんはどちらの修正を優先すべきか分からないので**コンフリクト**が起こってしまう。  
この時、Aが先にmainにbranch_Aをマージしていた場合、Bがコンフリクトを解消することになる。  
Bのコンフリクト解消のマニュアルを以下に書き記す。  
プルリク申請時に`Can't automatically merge`が表示されるがそのままプルリクを申請。  
プルリクを開き、下部にある`resolove conflicts`を押しコンフリクト解消を行っていく。  
コマンドラインでのコンフリクト解消は公式が[ドキュメント](https://docs.github.com/ja/pull-requests/collaborating-with-pull-requests/addressing-merge-conflicts/resolving-a-merge-conflict-on-github)を出しているのでそちらを見てください。  
コンフリクト解消をしたらそのままマージしましょう。  
以上でコンフリクト解消完了

### その他のコマンド
- リポジトリの状態を見る`git status`
- コミット履歴を見る`git log`
- ブランチ一覧`git branch`
- ブランチ移動`git checkout 移動したいブランチ名`
- 差分確認`git diff`

## 作業ブランチから新たな作業ブランチを切るのはご法度！！！！
 作業ブランチから新たな作業ブランチを切ると前のブランチに依存したブランチを切ることになるので**コンフリクト**(修正場所が被ること)が起こりやすい。
 新たに作業ブランチを切りたい場合は`git checkout main`してからにしましょう
